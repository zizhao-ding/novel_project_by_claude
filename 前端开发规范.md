## 1. 项目结构

### 1.1 目录结构

```
src/
├── assets/          # 静态资源（图片、图标、字体、全局样式）
├── components/      # 公共组件
│   ├── base/        #   基础组件（对 Element Plus 的二次封装）
│   ├── business/    #   业务组件（含业务逻辑的复合组件）
│   └── ui/          #   UI 组件（纯展示型布局组件）
├── views/           # 页面视图（每个路由对应一个视图）
├── router/          # 路由配置 + 路由守卫
├── stores/          # Pinia 状态管理
├── services/        # API 服务层（HTTP 请求封装）
├── composables/     # Vue Composables（可复用的响应式逻辑）
├── utils/           # 纯工具函数（无副作用、无响应式）
├── types/           # TypeScript 类型定义
├── directives/      # Vue 自定义指令
├── plugins/         # Vue 插件（全局错误处理等）
└── constants/       # 常量定义
```

### 1.2 组件分层理由

| 层级 | 目录 | 何时放入 | 举例 |
|------|------|----------|------|
| 基础组件 | `components/base/` | 对 UI 库组件的统一封装，屏蔽第三方 API 变化 | `BaseButton`, `BaseModal` |
| 业务组件 | `components/business/` | 包含项目特有业务逻辑，可跨页面复用 | `NovelCard`, `UploadPanel` |
| UI 组件 | `components/ui/` | 纯展示、无业务逻辑的布局组件 | `PageHeader`, `Sidebar` |
| 视图 | `views/` | 与路由一一对应，组合组件构成页面 | `HomeView`, `LibraryView` |

> **为什么分三层？** 基础组件隔离第三方依赖（换 UI 库时只改 base 层），业务组件封装复用逻辑，UI 组件保持纯净和通用。

### 1.3 文件命名

| 文件类型 | 命名方式 | 示例 |
|----------|----------|------|
| Vue 组件 | PascalCase | `UserCard.vue`, `NovelReader.vue` |
| JS/TS 工具 | kebab-case | `api-client.ts`, `format-date.ts` |
| 样式文件 | kebab-case | `variables.scss` |
| 测试文件 | `*.spec.ts` | `user-card.spec.ts` |

> **为什么组件用 PascalCase？** Vue 官方推荐，与 SFC 的组件注册机制一致，且区分于普通 JS 模块。

代码示例见：`.claude/rules/01_project_structure.md`

## 2. 技术选型决策

### 2.1 Vue 3 Composition API

**选择理由**：
- 更好的逻辑复用（Composables 替代 Mixins）
- 更好的 TypeScript 类型推断支持
- 更灵活的组织代码（按功能而非选项类型分组）
- `<script setup>` 语法糖减少样板代码

**禁止 Options API**：保持代码风格统一，避免同一项目中两种写法混杂。

代码示例见：`.claude/rules/02_vue3_components.md`

### 2.2 TypeScript

**选择理由**：
- 编译期捕获类型错误，减少运行时 Bug
- IDE 智能提示和自动补全
- 代码即文档（接口定义即数据结构说明）
- 与后端 FastAPI + Pydantic 的类型体系对齐

**约束**：所有新文件使用 `.ts`，所有 `.vue` 使用 `<script setup lang="ts">`。

代码示例见：`.claude/rules/03_typescript.md`

### 2.3 Pinia（非 Vuex）

**选择理由**：
- Vue 官方推荐的状态管理库（Vuex 已进入维护模式）
- Setup Store 模式与 Composition API 风格一致
- 天然支持 TypeScript 类型推导
- 无 Mutation 概念，API 更简洁
- 支持多个 Store 实例，模块化更自然

代码示例见：`.claude/rules/04_state_management.md`

### 2.4 Element Plus

**选择理由**：
- 国内最成熟的 Vue 3 UI 组件库，社区活跃
- 组件覆盖面广（表单、表格、弹窗、上传等）
- 配套图标库完善
- 支持按需导入，减少打包体积

**约束**：使用 `unplugin-vue-components` 按需导入，禁止引入其他 UI 库。

### 2.5 Axios 封装

**为什么封装而不直接使用 fetch？**
- 统一配置 baseURL、timeout、headers
- 请求拦截器自动附加 Token
- 响应拦截器统一处理 401、500 等错误
- 避免在每个组件中重复编写鉴权和错误处理逻辑

代码示例见：`.claude/rules/05_api_services.md`

### 2.6 BEM + UnoCSS + Element Plus 变量

**样式三层策略**：

| 层级 | 方案 | 用途 |
|------|------|------|
| 原子化 | UnoCSS | 日常布局（flex, p-4, text-center） |
| 组件样式 | SCSS + BEM | 复杂自定义样式 |
| 主题色 | Element Plus CSS 变量 | 统一使用 `--el-color-primary` 等变量 |

> **为什么用 BEM 而非 CSS Modules？** BEM 命名约定不依赖构建工具，调试时类名语义清晰，且与 SCSS 嵌套语法配合良好。

代码示例见：`.claude/rules/06_styling.md`

## 3. 架构约束

### 3.1 路由

- **必须懒加载**：所有视图组件使用 `() => import(...)`，减少首屏加载时间
- **路由守卫分层**：鉴权逻辑放在 `router/guards.ts`，不污染路由配置文件
- **Meta 驱动**：页面标题、权限要求等信息通过 `meta` 字段声明

代码示例见：`.claude/rules/07_routing.md`

### 3.2 错误处理

**分层处理策略**：

| 层级 | 处理范围 | 方式 |
|------|----------|------|
| 全局 | 未捕获的 Vue 错误 + Promise 拒绝 | `app.config.errorHandler` + `unhandledrejection` |
| HTTP | 401/500 等通用 HTTP 错误 | Axios 响应拦截器 |
| Store | 业务操作失败 | try-catch + error state |
| 组件 | 子组件错误边界 | `onErrorCaptured` |

> **为什么不只用全局错误处理？** 不同层级的错误需要不同的用户提示和恢复策略。全局兜底保证不会白屏，业务层给出具体操作引导。

代码示例见：`.claude/rules/08_error_handling.md`

### 3.3 数据流

```
组件 ←→ Store ←→ API Service ←→ 后端
  │        │
  └── composables（可复用逻辑）
```

- **组件**：纯视图逻辑，只负责渲染和事件绑定
- **Store**：全局状态 + 业务操作，组件通过 action 修改数据
- **API Service**：纯 HTTP 调用，不含业务逻辑
- **Composables**：横切关注点（如 `useAuth`, `usePagination`）

## 5. Git 规范

### 5.1 分支策略

```
main                    # 生产分支
├── feature_1.0.0.1     # 版本集成分支
│   └── feature_zizhao  # 个人功能分支
```

### 5.2 Commit 格式

```
<type>(<scope>): <subject>
```

类型：`feat` | `fix` | `docs` | `style` | `refactor` | `perf` | `test` | `chore`

代码示例见：`.claude/rules/09_git_conventions.md`
