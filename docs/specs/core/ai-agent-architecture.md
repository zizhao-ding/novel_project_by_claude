# AI Agent 原生六层架构

> **定位**: 项目架构总纲 — AI Agent 的治理框架
> **优先级**: P0 — 所有 AI 行为的基础
> **创建日期**: 2026-06-30
> **替代**: Harness 六层架构（内容已融合进新框架）

---

## 1. 设计哲学

### 从流水线到并行治理

传统软件工程架构（如 Harness）是**流程导向**的：

```
需求层 → 规范层 → 方案层 → 执行层 → 审核层 → 验收层
```

一条线性流水线从需求走到交付，每个阶段有明确的输入输出和门控。这对人类团队管理项目很有用，但对 AI Agent 来说不够——AI 不是流水线工人，它是**持续的工程伙伴**。

AI Agent 在每一次行为（生成代码、修改文件、运行命令）时，需要同时受到多个维度的约束：

- 这段代码符合规范吗？（质量维度）
- 这个操作在当前任务范围内吗？（上下文维度）
- 有没有触犯红线？（约束维度）
- 历史经验告诉我们应该怎么做？（记忆维度）
- 我有权限做这个吗？（工具维度）
- 当前处于什么流程阶段？（流程维度）

因此，AI Agent 原生架构采用**多维度并行治理**模型——六个维度始终同时生效。

### 核心理念

```
传统架构：时间维度（先做什么，再做什么）
AI 架构：空间维度（每次都检查什么）
```

## 3. 六层详解

### 第一层：规范层 (Specification Layer)

> **"代码应该长什么样？"**

规范层定义了 AI 的**审美标准**——什么是好代码，什么是坏代码。

| 属性 | 说明 |
|------|------|
| **性质** | 建议性标准（违反也能跑，但质量下降） |
| **生效时机** | 生成代码时查阅，审查代码时对照 |
| **更新频率** | 低（项目初期确立，稳定运行） |

**内容清单**：

| 文档 | 路径 | 覆盖范围 |
|------|------|---------|
| 编码规范 | `docs/specs/core/coding-standards.md` | 文件命名、Vue组件模板、TypeScript类型、SCSS/BEM、Git提交 |
| 架构规范 | `docs/specs/core/architecture.md` | Pinia Store、API服务层、路由设计、错误处理、敏感操作确认 |
| 详细规范 | `docs/specs/detail/` | 具体实现指南 |
| 模块规范 | `docs/specs/module/` | 特定模块的开发规范 |

**与其他层的关系**：
- 约束层定义**禁止什么**，规范层定义**推荐怎么做**——前者是红线（硬），后者是方向（软）
- 违反规范层 → 代码审核时警告
- 违反约束层 → 阻止执行

---

### 第二层：上下文层 (Context Layer)

> **"现在在做什么？做到哪了？"**

上下文层是 AI 的**工作记忆**——避免每次新会话都从零开始。

| 属性 | 说明 |
|------|------|
| **性质** | 动态状态（随项目进展持续变化） |
| **生效时机** | 会话启动时加载，会话结束时更新 |
| **更新频率** | 高（每次开发会话后更新进度） |

**内容清单**：

| 文档 | 路径 | 提供的信息 |
|------|------|-----------|
| 项目总索引 | `docs/index.md` | 项目结构、技术栈、快速命令 |
| 需求索引 | `docs/requirements/index.md` | 所有需求的状态（pending/developing/done） |
| 当前进度 | `docs/ai-memory/global/03_current-progress.md` | 已完成/进行中/待开始的 checklist |
| 本文件 | `CLAUDE.md` | 项目概述 + 会话级上下文 |

**会话启动时的上下文加载顺序**：

```
1. docs/index.md                → 了解项目全貌
2. docs/requirements/index.md   → 了解需求状态
3. docs/ai-memory/global/03_current-progress.md → 了解当前进度
4. docs/ai-memory/global/06_architecture-decisions.md → 了解关键决策
```

**与其他层的关系**：
- 上下文层告诉 AI "我们在哪"——没有它，规范层和约束层都是无目标的
- 工作流层的启动流程定义了上下文加载的顺序
- 记忆层为上下文层提供历史脉络

---

### 第三层：约束层 (Constraint Layer)

> **"什么绝对不能做？"**

约束层是 AI 的**宪法**——不可违反的硬性红线。

| 属性 | 说明 |
|------|------|
| **性质** | 硬性规则（违反即错误，不可协商） |
| **生效时机** | 每次代码生成前（PreToolUse）和后（PostToolUse）自动检查 |
| **更新频率** | 极低（只有重大技术决策变更时才调整） |

**约束分类**：

| 类别 | 约束项 | 来源 |
|------|--------|------|
| 🔴 技术栈 | 必须使用 TypeScript（禁止纯 JS） | 编码规范 |
| 🔴 技术栈 | 必须使用 Element Plus（禁止其他 UI 库） | 编码规范 |
| 🔴 技术栈 | 必须使用 Composition API + `<script setup>`（禁止 Options API） | 编码规范 |
| 🔴 技术栈 | 必须使用 Pinia Setup Store（禁止 Options Store） | 架构规范 |
| 🔴 代码 | 禁止 `var`、`==`（使用 `const`/`let`、`===`） | 编码规范 |
| 🔴 代码 | 禁止 `any`（使用 `unknown` + 类型守卫） | 编码规范 |
| 🔴 代码 | 禁止在模板中使用复杂表达式 | 编码规范 |
| 🔴 代码 | 禁止直接操作 DOM | 编码规范 |
| 🔴 安全 | 禁止 `v-html` 直接渲染用户输入 | 安全规范 |
| 🔴 安全 | 删除操作必须有二次确认 | 架构规范 |
| 🔴 流程 | 提交前必须通过 ESLint + TypeScript 检查 | Hook 规则 |
| 🔴 流程 | 敏感操作必须有用户确认 | Hook 规则 |

**详细定义**：`docs/specs/core/constraint-layer.md`

**与其他层的关系**：
- 约束层是规范层的"底线版"——规范说"应该"，约束说"必须"或"禁止"
- Hook 规则（`hook-rules.md`）是约束层的**执行机制**——PreToolUse 和 PostToolUse 是约束落地的技术手段
- 约束层 > 规范层：当两者冲突时，约束层优先

---

### 第四层：记忆层 (Memory Layer)

> **"过去学到了什么？"**

记忆层是 AI 的**长期记忆**——让 AI 随项目成长而持续变聪明。

| 属性 | 说明 |
|------|------|
| **性质** | 持续积累（每次开发完成后沉淀） |
| **生效时机** | 会话启动时回顾相关记忆，开发中遇到相似场景时查阅 |
| **更新频率** | 中（每次功能完成后可能新增记忆） |

**内容清单**：

| 类别 | 路径 | 内容 |
|------|------|------|
| 架构决策记录 | `docs/ai-memory/global/06_architecture-decisions.md` | 为什么做了某个设计选择（5条ADR） |
| 技术债务 | `docs/ai-memory/global/05_testing-debt.md` | 测试覆盖率现状和改进计划 |
| 项目经验 | `docs/ai-memory/global/` 其他文件 | 项目级别的经验教训 |
| 模块经验 | `docs/ai-memory/module/` | 特定模块的经验（认证、上传等） |

**记忆的生命周期**：

```
开发中遇到问题 → 解决 → 沉淀为记忆 → 下次会话自动加载 → 指导类似决策
```

**已记录的关键记忆**：

| ADR | 决策 | 日期 |
|-----|------|------|
| ADR-001 | 权限系统采用纯可见性控制，书架回归收藏夹 | 2026-06-27 |
| ADR-002 | 全局统一 AppHeader 替代各页面自行实现 | 2026-06-28 |
| ADR-003 | 阅读器三主题 + 进度持久化 | 2026-06-28 |
| ADR-004 | 二次确认弹窗在视图层而非 Store 中 | 2026-06-26 |
| ADR-005 | 后端权限采用函数式 Depends 注入 | 2026-06-27 |

**与其他层的关系**：
- 记忆层是**旧 Harness 最大的盲区**——Harness 有记忆目录但地位模糊
- 上下文层回答"现在是什么状态"，记忆层回答"为什么是这个状态"
- 工作流层的"方案设计"阶段应主动查阅记忆层的相关 ADR

---

### 第五层：工具层 (Tool Layer)

> **"AI 能调用什么？权限边界在哪？"**

工具层定义了 AI 的**能力圈**——明确知道自己的工具箱和权限。

| 属性 | 说明 |
|------|------|
| **性质** | 能力声明（不是工具的实现，而是工具的选择规范） |
| **生效时机** | 每次选择调用哪个工具时 |
| **更新频率** | 低（跟随 Claude Code 版本更新） |

**工具清单与使用场景**：

| 工具 | 推荐场景 | 需要确认 |
|------|---------|---------|
| `Read` | 读取文件内容 | 否 |
| `Write` | 创建或覆盖文件 | 否（项目内） |
| `Edit` | 精确修改文件片段 | 否（项目内） |
| `Bash` | 执行命令（构建、lint、git） | 取决于操作 |
| `Glob` | 按模式查找文件 | 否 |
| `Grep` | 搜索代码内容 | 否 |
| `Agent` | 委托复杂任务给子 Agent | 否 |
| `WebFetch` | 获取外部网页内容 | 否 |
| `WebSearch` | 搜索互联网 | 否 |
| `TaskCreate` | 创建任务跟踪 | 否 |
| `AskUserQuestion` | 需要用户决策时 | 否 |

**权限边界**：

```
┌─────────────────────────────────────────┐
│  AI 可以自主执行                          │
│  - 读取项目文件                           │
│  - 修改项目文件（非敏感操作）              │
│  - 运行 lint/type-check                  │
│  - 搜索代码库                             │
│  - 创建文档                               │
├─────────────────────────────────────────┤
│  需要用户确认                             │
│  - 删除文件/目录                          │
│  - 修改配置文件                           │
│  - 执行 git push                          │
│  - 安装新依赖                             │
│  - 运行危险命令                           │
├─────────────────────────────────────────┤
│  AI 不能执行                              │
│  - 强制推送到共享分支                     │
│  - 绕过 Hook 检查提交代码                 │
│  - 忽略约束层的硬性规则                   │
└─────────────────────────────────────────┘
```

**工具使用规范**：

- ✅ 读文件优先使用 `Read`，非 `cat` 命令
- ✅ 搜索文件优先使用 `Glob`，非 `ls` 或 `find`
- ✅ 搜索代码优先使用 `Grep`，非 `grep` 命令
- ✅ 复杂多文件任务使用 `Agent` 工具并行处理
- ❌ 禁止在 `Bash` 中内联大段代码（应使用 `Write` 写入文件）
- ❌ 禁止不经验证直接执行用户提供的命令

**与其他层的关系**：
- 约束层的 Hook 规则定义了工具调用前后的检查
- 工作流层定义了"在什么阶段用什么工具"的模式
- 工具层是六层中唯一**直接对接外部系统**（Claude Code）的层

### 第六层：工作流层 (Workflow Layer)

> **"任务按什么流程执行？"**

工作流层是 AI 的**SOP 手册**——定义了多条标准执行流程。

| 属性 | 说明 |
|------|------|
| **性质** | 执行编排（定义步骤顺序和阶段门控） |
| **生效时机** | 每次开始一个任务时选择合适的工作流 |
| **更新频率** | 中（新的任务类型出现时新增工作流） |

**工作流清单**：

#### 流程 A：会话启动流程

```
1. 读取 docs/index.md              → 了解项目全貌
2. 读取 docs/requirements/index.md → 获取需求状态
3. 读取 约束层文档                  → 加载硬性红线
4. 读取 记忆层关键文件              → 回顾经验教训
   - 03_current-progress.md        → 当前进度
   - 06_architecture-decisions.md  → 架构决策
   - 05_testing-debt.md            → 技术债务
5. 确定本次会话目标                 → 从待处理需求中选择
```

#### 流程 B：标准开发流程

```
需求分析 → 方案设计 → 代码实现 → 审核检查 → 验收完成
   │          │          │          │          │
   │          │          │          │          └── 更新需求状态 + 沉淀记忆
   │          │          │          └── Lint + TypeCheck + CodeReview
   │          │          └── 遵循规范层 + 约束层实时检查
   │          └── 查阅记忆层 ADR + 设计技术方案
   └── 加载上下文层相关需求文档
```

#### 流程 C：快速修复流程

```
1. 定位问题（Grep/Read 搜索相关代码）
2. 确认修复方案（检查约束层：修复方案是否违反红线？）
3. 实施修复（Edit 精确修改）
4. 验证修复（Bash: npm run lint && npm run type-check）
5. 完成（无需更新需求文档，但重要的修复应记录到记忆层）
```

#### 流程 D：代码审查流程

```
1. 加载被审查的代码
2. 对照规范层逐项检查（命名、结构、类型）
3. 对照约束层检查红线
4. 对照记忆层检查是否重复已知错误
5. 输出审查报告（严重/建议/夸奖）
```

#### 流程 E：会话结束流程

```
1. 更新 03_current-progress.md     → 同步 checkbox 状态
2. 更新需求文档状态标签             → pending/developing/done
3. 新增记忆（如有值得记录的经验）   → 沉淀到记忆层
4. 如有重大决策 → 补充 ADR         → 记录到 06_architecture-decisions.md
5. 更新 docs/index.md 中的需求列表  → 保持同步
```

**与其他层的关系**：
- 工作流层融合了旧 Harness 的流水线思想（L1→L6），但更灵活——多流程而非单流水线
- 每个工作流阶段都要查询上下文层、遵守规范层、检查约束层、查阅记忆层
- 工作流层是六层的"调度中心"——它编排其他五层在时间维度上的配合

## 4. 六层协同机制

六层不是独立的——它们在 AI 的每一次行为中**同时生效**。以"添加删除评论功能"为例：

```
用户请求："给评论列表加个删除功能"

上下文层激活 ─── 当前在评论组件？涉及哪些文件？
    │
约束层激活 ─── 删除操作需要二次确认弹窗
    │          v-html 渲染用户评论内容是否安全？
    │
规范层激活 ─── 组件命名 CommentItem.vue
    │          SCSS 用 BEM：.comment-item__delete-btn
    │          handleDelete 方法放在视图层（ADR-004）
    │
记忆层激活 ─── ADR-004：确认弹窗在视图层而非 Store
    │          ADR-001：权限检查用 usePermission composable
    │
工具层激活 ─── Read 相关文件 → Edit 精确修改 → Bash run lint
    │         需要二次确认的操作需要 AskUserQuestion？
    │
工作流层激活 ── 走"小功能开发流程"（方案→执行→审核）
    │          会话结束时更新进度状态
```

## 6. 与 Harness 的映射关系

新架构**融合**了 Harness 的全部内容，而非丢弃：

| Harness 层级 | 融入了新架构的... | 说明 |
|-------------|------------------|------|
| L1 需求层 | 上下文层 + 记忆层 | 需求状态归上下文层（动态），需求历史归记忆层（沉淀） |
| L2 规范层 | 规范层 + 约束层 | 编码规范归规范层（建议），Hook 规则归约束层（强制） |
| L3 方案层 | 工作流层 + 记忆层 | 方案设计是工作流的一个阶段，决策记录归记忆层 |
| L4 执行层 | 工具层 | 代码生成/文件操作/构建 → 工具层定义的能力 |
| L5 审核层 | 工作流层 + 约束层 | 审核是工作流的一个阶段，审核标准来自规范层和约束层 |
| L6 验收层 | 工作流层 + 上下文层 | 验收是工作流的一个阶段，验收结果更新上下文层状态 |

**关键改进**：

| Harness 的不足 | 新架构的改进 |
|---------------|------------|
| 单向流水线，缺少反馈闭环 | 六层并行治理，每次行为都经过六维检查 |
| 上下文散落各处，缺乏统一管理 | 上下文层独立，会话启动时统一加载 |
| 约束混在规范中，"红线"不清晰 | 约束层独立，硬性红线一目了然 |
| 记忆目录地位模糊 | 记忆层是一等层级，持续积累经验 |
| 只有一条流程（需求→交付） | 工作流层支持多条 SOP |
| 工具能力未显式定义 | 工具层明确能力边界和权限 |

## 7. 快速参考

### AI 收到请求后的检查清单

```
□ 上下文层：我知道当前项目状态和目标吗？
□ 约束层：这个操作有没有触犯硬性红线？
□ 规范层：生成的代码符合质量标准吗？
□ 记忆层：有没有相关历史经验可以借鉴？
□ 工具层：用的工具对吗？需要用户确认吗？
□ 工作流层：当前处于什么流程阶段？下一步是什么？
```

### 各层负责人（AI 视角）

| 层 | AI 的责任 |
|----|----------|
| 规范层 | 生成代码前查阅规范，审查代码时对照规范 |
| 上下文层 | 会话开始时加载状态，会话结束时更新状态 |
| 约束层 | 每次工具调用前自动检查 Hook 规则 |
| 记忆层 | 遇到相似场景时主动查阅相关记忆 |
| 工具层 | 选择最合适的工具，遵守权限边界 |
| 工作流层 | 根据任务类型选择合适的流程 |
