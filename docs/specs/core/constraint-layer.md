# 约束层 — 硬性红线

> **定位**: AI Agent 六层架构 — 第三层
> **性质**: 硬性规则，不可违反，不可协商
> **优先级**: P0 — 违反即阻止执行
> **创建日期**: 2026-06-30

---

## 概述

约束层定义了 AI Agent 的**宪法**。这些规则不是建议、不是推荐——它们是**绝对的红线**。

| 规范层（建议） | 约束层（红线） |
|-------------|-------------|
| "应该用 BEM 命名" | "必须用 TypeScript" |
| "推荐用 computed 替代模板表达式" | "禁止用 Options API" |
| 违反 → 代码审核警告 | 违反 → 阻止执行 |

**约束来源**：编码规范、架构规范、Hook 规则、安全规范——从这些文档中提取所有 `必须` 和 `禁止` 项。

**执行机制**：Hook 规则（`hook-rules.md`）是约束层的技术执行手段——PreToolUse 在代码生成前检查，PostToolUse 在代码生成后验证。

## 约束清单

### 🔴 C-001 ~ C-005: 技术栈强制

| 编号 | 约束 | 说明 |
|------|------|------|
| C-001 | **必须使用 TypeScript** | 所有 `.vue` 文件 `<script setup lang="ts">`，工具文件 `.ts` 扩展名，禁止纯 JavaScript |
| C-002 | **必须使用 Element Plus** | 唯一 UI 组件库，禁止引入 Ant Design、Naive UI 等其他 UI 库 |
| C-003 | **必须使用 Composition API** | 所有组件使用 `<script setup>` + Composition API（`ref`, `computed`, `watch` 等），禁止 Options API（`data()`, `methods`, `computed` 选项） |
| C-004 | **必须使用 Pinia Setup Store** | `defineStore('name', () => { ... })` 模式，禁止 Options Store（`state: () => ({...})`） |
| C-005 | **路由必须懒加载** | 所有页面组件使用 `() => import(...)`，禁止直接 import |

### 🔴 C-006 ~ C-010: 代码质量底线

| 编号 | 约束 | 说明 |
|------|------|------|
| C-006 | **禁止 `var`** | 使用 `const` 或 `let` |
| C-007 | **禁止 `==`** | 使用 `===`（严格相等） |
| C-008 | **禁止 `any`** | 使用 `unknown` + 类型守卫，或正确定义类型。极端情况下用 `@ts-expect-error` 并附说明，禁止 `@ts-ignore` |
| C-009 | **禁止模板复杂表达式** | 抽取为 `computed` 或方法 |
| C-010 | **禁止直接操作 DOM** | 使用 Vue 响应式绑定（`ref`、`v-model`），禁止 `document.querySelector` 等 |

### 🔴 C-011 ~ C-015: 架构约束

| 编号 | 约束 | 说明 |
|------|------|------|
| C-011 | **禁止组件中直接使用 axios/fetch** | 统一通过 `services/` 层的 `api.client` 发送请求 |
| C-012 | **禁止硬编码 API URL** | 使用环境变量或配置 |
| C-013 | **Store 中禁止调用 UI 组件** | Store 保持纯数据逻辑，`ElMessageBox`、`ElMessage` 等 UI 交互放在视图层（ADR-004） |
| C-014 | **禁止跨层级相对路径导入** | 使用 `@` 别名（`@/components/...`），非 `../../../components/...` |
| C-015 | **禁止 Store state 直接修改** | 必须通过 Store 的 action 方法修改 |

### 🔴 C-016 ~ C-020: 安全约束

| 编号 | 约束 | 说明 |
|------|------|------|
| C-016 | **删除操作必须有二次确认** | 使用 `ElMessageBox.confirm` 弹窗，确认后才执行 |
| C-017 | **禁止 `v-html` 渲染用户输入** | XSS 风险。必须使用时需经过 DOMPurify 等消毒处理 |
| C-018 | **所有异步操作必须有错误处理** | 使用 `try-catch-finally`，禁止空 `catch` 块吞掉错误 |
| C-019 | **错误信息不能直接暴露给用户** | 原始错误对象不能直接显示，需要转换为用户友好的提示 |
| C-020 | **敏感操作（退出/覆盖/发布）需二次确认** | 同 C-016，范围更广 |

### 🔴 C-021 ~ C-025: 文件结构约束

| 编号 | 约束 | 说明 |
|------|------|------|
| C-021 | **Vue 组件文件名 PascalCase** | `UserCard.vue`，非 `user-card.vue` 或 `usercard.vue` |
| C-022 | **视图文件以 `View` 结尾** | `HomeView.vue`，非 `Home.vue` 或 `HomePage.vue` |
| C-023 | **文件必须在正确的目录** | 组件在 `components/`，Store 在 `stores/`，工具在 `utils/`，类型在 `types/` |
| C-024 | **禁止在 `src/` 根目录创建业务文件** | 必须归入对应的子目录 |
| C-025 | **禁止提交 `console.log` 调试代码** | 开发完成前清理所有调试输出 |

### 🔴 C-026 ~ C-030: 流程约束

| 编号 | 约束 | 说明 |
|------|------|------|
| C-026 | **提交前必须通过 ESLint** | `npm run lint` 零错误 |
| C-027 | **提交前必须通过 TypeScript 检查** | `npm run type-check` 零错误（等效于 `vue-tsc --noEmit`） |
| C-028 | **提交信息必须符合 Conventional Commits** | `<type>(<scope>): <subject>` 格式 |
| C-029 | **禁止 `git push --force` 到共享分支** | `main` 和集成分支 |
| C-030 | **会话结束必须更新进度记录** | 更新 `03_current-progress.md` 和需求文档状态 |
| C-031 | **问题解决后主动询问是否 commit** | 由用户决定提不提交，不自动 commit |
| C-032 | **Commit 前必须通过 Lint + TypeCheck** | 同 C-026/027，每次 commit 前自动执行 |

## 约束优先级

当约束之间发生冲突时，按以下优先级裁决：

```
安全约束 > 架构约束 > 技术栈约束 > 代码质量约束 > 文件结构约束 > 流程约束
```

示例：
- 如果一个"正确的文件结构"导致了 XSS 风险 → 安全约束胜出
- 如果一个"代码质量建议"和"架构约束"冲突 → 架构约束胜出

## AI 行为准则

### 收到代码生成请求时

```
1. 逐条检查 C-001 ~ C-025
2. 任何一条触发 → 停止生成，先修正方案
3. 全部通过 → 继续生成
```

### 收到工具调用请求时

```
1. 检查操作是否触发 PRE-001 ~ PRE-004
2. 触发且未满足条件 → 阻止调用，先修正
3. 满足条件 → 允许调用
```

### 提交代码前

```
1. 运行 npm run lint        → 检查 C-001/003/006~010/021~025
2. 运行 npm run type-check  → 检查 C-001/008
3. 检查 commit message      → 检查 C-028
```
